# AI WUXING SKILL BIBLE

## Part I — Philosophy & Visual Design Bible

> This document defines the artistic philosophy and visual language that every AI-generated skill MUST follow.
>
> The purpose is **NOT** merely to create functional skills, but to create memorable, satisfying, premium-quality visual experiences comparable to modern AAA action games.
>
> API documentation explains **how** to build a skill.
>
> This document explains **how to make players remember it.**

---

# 1. Core Philosophy

Every skill should answer four questions.

1. What is happening?
2. How powerful is it?
3. What element does it belong to?
4. What is the most memorable moment?

If a player cannot answer these questions within half a second, the visual design has failed.

A beautiful skill is not the one with the most particles.

A beautiful skill is the one that communicates power, identity, timing and impact instantly.

---

# 2. Design Priorities

When designing a skill, prioritize these goals in order.

```
Readability
↓

Identity
↓

Motion

↓

Impact

↓

Detail

↓

Performance
```

Never sacrifice readability for visual complexity.

Players should always understand:

* where the skill starts
* where it goes
* where it will hit
* when it becomes dangerous

---

# 3. Every Skill is a Story

Never think of a skill as

```
Spawn Projectile

Explosion
```

Think instead as

```
Preparation

↓

Energy Gathering

↓

Release

↓

Travel

↓

Impact

↓

Aftermath

↓

Fade Out
```

A premium skill always tells a visual story.

---

# 4. The Six Visual Beats

Every skill should be composed of these six phases.

---

## Phase 1 — Anticipation

Purpose

Prepare the player's eyes.

Examples

* weapon charging
* magic circle appears
* particles gathering
* glow increasing
* ribbon forming
* character pose
* energy compression

Typical duration

```
10%–20%
```

---

## Phase 2 — Release

Purpose

Maximum acceleration.

Examples

* sword leaves hand
* fire erupts
* beam starts
* dragon emerges
* lightning strikes

This phase should feel explosive.

---

## Phase 3 — Travel

Purpose

Maintain visual interest.

Never allow a projectile to simply fly.

During travel add

* trail
* sparks
* ribbon
* wobble
* smoke
* dynamic light
* distortion
* secondary particles

Movement should evolve continuously.

---

## Phase 4 — Impact

Purpose

Maximum visual climax.

This is the strongest frame.

Everything peaks here.

Examples

* flash
* shockwave
* explosion
* sparks
* camera shake
* screen distortion
* debris
* ground crack

---

## Phase 5 — Aftermath

Purpose

Show consequences.

Examples

* smoke
* mist
* burning ground
* glowing cracks
* floating debris
* fading ribbon
* residual lightning

Never disappear immediately after impact.

---

## Phase 6 — Dissolve

Purpose

Smoothly return the screen to calm.

Objects should

* fade
* dissolve
* shrink
* evaporate
* burn away
* scatter

Avoid abrupt disappearance.

---

# 5. Hero Moment

Every skill MUST contain one Hero Moment.

Hero Moment means

> The single frame players remember.

Examples

Good Hero Moments

* giant sword splitting the battlefield
* dragon opening its mouth
* tornado reaching maximum size
* massive lightning convergence
* mountain erupting
* water vortex collapsing

Bad Hero Moments

* random particle burst
* generic explosion
* sphere becoming larger

One Hero Moment is worth more than thousands of particles.

# 8. Motion Design

Nothing should move at constant speed.

Everything should contain motion curves.

Preferred motion

```
Ease Out

Ease In

Overshoot

Bounce

Oscillation

Settling
```

Avoid

```
Constant Velocity

Instant Stop

Linear Fade
```

Natural motion feels alive.

Robotic motion feels fake.

---

Avoid uniform density.

Instead create contrast.

This feels natural.


Players should understand a skill instantly.

* overlapping colors
* excessive bloom

Preferred sequence

Examples

---

# 19. Organic Design

Nature is imperfect.

Every generated object should contain variation.

Randomize

* rotation
* scale
* spacing
* timing
* curvature
* wobble
* noise

Avoid perfect grids.

Avoid perfectly straight lines.

Avoid identical particles.

Imperfection creates realism.

---

# 20. The 80/20 Rule

Spend most visual budget on

* anticipation
* impact
* hero moment

Do not waste thousands of particles during travel.

Players remember beginnings and endings.

---

# 21. Common Mistakes

Never create

* perfect sphere projectile
* straight particle line
* static laser
* generic explosion
* identical particles
* constant-speed movement
* instant disappearance
* random visual spam

These effects look procedural instead of handcrafted.

---

# 22. Final Visual Checklist

Before finishing any skill, verify:

□ Clear anticipation

□ Strong release

□ Interesting travel

□ Memorable hero moment

□ Powerful impact

□ Meaningful aftermath

□ Multiple VFX layers

□ Secondary motion

□ Dynamic lighting

□ Camera feedback

□ Ground interaction

□ Large / Medium / Small contrast

□ Organic randomness

□ Strong silhouette

□ Consistent elemental identity

□ Excellent gameplay readability

If any checkbox is missing, continue improving the skill.

A skill is finished not when there is nothing more to add,

but when every visual element contributes to gameplay clarity, elemental identity, and emotional impact.

# AI WUXING SKILL BIBLE

# Part III — Engine Capability & Best Practices

> This chapter explains **what the engine can do**, **when each system should be used**, and **how to combine systems** to create premium-quality skills.
>
> **Do not think in terms of individual APIs.**
>
> Think in terms of **visual systems working together**.

---

# 1. Design Philosophy

The engine is composed of independent systems.

Every system has one responsibility.

Never force one system to perform another system's job.

| System      | Responsibility       |
| ----------- | -------------------- |
| Skill       | Gameplay logic       |
| Trail       | Hero object movement |
| Particle    | Fine visual details  |
| Force Field | Procedural motion    |
| Mesh        | Physical geometry    |
| Shader      | Rendering only       |
| Camera      | Feedback             |
| Lighting    | Visual emphasis      |

A beautiful skill is built by combining systems, **not by overloading one system**.

---

# 2. General Layer Order

Recommended rendering and design order.

```

↓

Hero Mesh

↓

Trail

↓

Particles

↓

Force Field

↓

Lighting

↓

↓

Camera

↓

```

Each layer should have a clear purpose.

---

# 3. Skill System

The Skill System controls gameplay.

Responsibilities

* damage
* cooldown
* timing
* collision
* targeting
* state machine
* spawning visual systems

The Skill System should **never** directly implement rendering.

It should only coordinate other systems.

---

Meshes define the physical object.

Examples

* sword
* spear
* dragon
* rock
* meteor
* beam
* shield

A mesh should represent the hero object.

Never use particles to fake an important object if a mesh exists.

Good

```

Trail

+

Particles
```

Bad

```

```

Trail is responsible for **motion visualization**.

Never use Trail to replace particles.

Never use Trail to fake explosions.

Visual quality is not determined by particle count.

* layering
* composition
* motion
* timing
* lighting
* silhouettes

❌ Use particles as the main object.

❌ Use trails to create explosions.

❌ Use force fields to fake gameplay.

❌ Spawn thousands of particles to hide poor composition.

❌ Make every layer equally bright.

❌ Spawn every effect at exactly the same frame.

❌ Let every visual disappear simultaneously.

---

# 18. The Golden Rule

Every system has a purpose.

```
Gameplay

controls logic.

Mesh

defines form.

Trail

shows motion.

Particle

adds detail.

Force Field

creates life.

Lighting

creates focus.

Camera

creates feedback.
```

When every system performs its own responsibility, the resulting skill becomes cleaner, easier to maintain, more reusable, and visually richer.

A premium skill is not built by making one system do everything.

It is built by allowing many specialized systems to work together in harmony.

# AI WUXING SKILL BIBLE

# Part IV — Skill Architecture & Gameplay Design

> This chapter defines **how an AI should architect an entire skill**, not merely create visual effects.
>
> A good skill is not a collection of particles.
>
> It is a coordinated sequence of gameplay, animation, VFX, timing, scaling, and player feedback.

---

# 1. Skill Architecture Philosophy

Every skill is a collection of independent systems working together.

Never create a skill as one giant block of code.

Think in modules.

```text
Skill

├── Gameplay Logic
├── Visual Timeline
├── Hero Objects
├── Particle Systems
├── Trail Systems
├── Force Fields
├── Camera Feedback
├── Lighting
├── Sound
└── Cleanup
```

Each module should have only one responsibility.

---

# 2. Skill Lifecycle

Every skill MUST follow the same lifecycle.

```text
Idle

↓

Cast Begin

↓

Charge

↓

Release

↓

Travel

↓

Hit Detection

↓

Impact

↓

Aftermath

↓

Cleanup
```

Never skip phases.

Even an instant attack should internally follow this lifecycle.

---

# 3. Cast Phase

Purpose

Inform the player that something is about to happen.

Possible actions

* play animation
* rotate character
* spawn charging particles
* begin light buildup
* create magic circle
* play charging sound

No damage should occur here.

---

# 4. Charge Phase

Energy accumulates.

Typical effects

* increasing glow
* orbiting particles
* rotating trails
* vibrating weapon
* expanding magic circle
* converging sparks

Charge should increase visual tension.

---

# 5. Release Phase

This is the transition from stored energy into motion.

Typical actions

* projectile spawn
* beam activation
* sword launch
* dragon summon
* explosion start

Release should feel instantaneous.

Avoid slow starts unless intentionally designed.

---

# 6. Travel Phase

Travel is often the longest phase.

Never allow the projectile to simply move.

Instead continuously evolve it.

Examples

Projectile

↓

Trail grows

↓

Ribbon oscillates

↓

Particles emitted

↓

Light pulses

↓

Smoke expands

↓

Force Field modifies movement

Movement should always feel alive.

---

# 7. Hit Detection

Gameplay happens here.

Responsibilities

* collision
* damage
* knockback
* status effects
* penetration
* bouncing
* chaining

Visual systems should react to gameplay,

not replace it.

---

# 8. Impact Phase

Everything reaches maximum intensity.

Recommended order

```text
Flash

↓

Camera Shake

↓

Explosion

↓

Shockwave

↓

Debris

↓

Particles

↓

Ground Interaction

↓

Residual Effects
```

The strongest frame of the skill should occur here.

---

# 9. Aftermath Phase

Show the consequences.

Examples

* smoke
* burning ground
* floating debris
* glowing cracks
* lingering mist
* fading energy

Powerful skills always leave evidence behind.

---

# 10. Cleanup Phase

Every object should disappear gracefully.

Preferred methods

* dissolve
* fade
* shrink
* burn away
* evaporate
* scatter

Never instantly destroy visible objects.

---

# 11. State Machine

Each skill should internally behave as a state machine.

```text
Idle

↓

Preparing

↓

Charging

↓

Casting

↓

Travelling

↓

Colliding

↓

Impact

↓

Ending

↓

Finished
```

Never mix logic from different states.

Each state should perform only its own responsibilities.

---

# 12. Skill Scaling Philosophy

Increasing skill level should not only increase damage.

It should increase spectacle.

Every level should improve one or more of

* visual complexity
* motion
* layering
* interaction
* hero moments

Players should feel stronger even before reading numbers.

---

# 13. Level Scaling

Example

Level 1

Single sword

Level 2

Larger trail

Level 3

Additional sparks

Level 4

Ground interaction

Level 5

Secondary explosion

Avoid simply multiplying particle count.

---

# 14. Milestone Scaling

Milestones unlock new visual stages.

Example

Milestone 0

Normal projectile

Milestone 1

Projectile gains orbiting shards

Milestone 2

Impact creates shockwave

Milestone 3

Summons secondary swords

Milestone 4

Leaves persistent battlefield effect

Each milestone should feel transformative.

---

# 15. Quantity Scaling

Quantity should introduce formation changes.

Avoid

```text
5 identical projectiles
```

Prefer

```text
One large projectile

↓

Three orbiting projectiles

↓

Five converging projectiles

↓

Radial formation

↓

Sword rain
```

Quantity should increase visual intelligence,

not repetition.

---

# 16. Size Scaling

When size increases,

every dependent system should scale consistently.

Affected systems

* mesh
* trail width
* particle size
* particle emission
* light radius
* impact radius
* decal size
* ground crack
* camera shake

Never scale only the mesh.

---

# 17. Lifetime Scaling

Powerful skills should last longer.

Weak Skill

0.3–0.6 s

Medium Skill

0.8–1.2 s

Ultimate

1.5–3 s

Persistent battlefield effects may last even longer.

---

# 18. Spawn Timing

Never spawn everything simultaneously.

Recommended sequence

```text
Hero Object

↓

Trail

↓

Light

↓

Particles

↓

Smoke

↓

Residual Effects
```

Staggered timing creates richness.

---

# 19. Layer Activation

Each layer should activate independently.

Example

```text
Frame 0

Weapon appears

Frame 2

Trail begins

Frame 4

Glow increases

Frame 6

Particles emitted

Frame 8

Dynamic Light

Frame 10

Shockwave
```

Layer staggering creates cinematic motion.

---

# 20. Progressive Complexity

As the skill evolves,

visual complexity should increase.

Example

Beginning

Simple

↓

Travel

More effects

↓

Impact

Maximum complexity

↓

Aftermath

Simplification

This creates natural pacing.

---

# 21. Environmental Interaction

Powerful skills affect the world.

Possible interactions

* dust
* grass movement
* falling leaves
* burning terrain
* cracked earth
* splashing water
* floating debris

Environment reinforces power.

---

# 22. Visual Persistence

Not everything should disappear immediately.

Examples

Flash

0.05 s

Trail

0.4 s

Smoke

1.5 s

Ground crack

4 s

Burning ground

6 s

Lingering effects make the battlefield feel alive.

# 25. Reusability

Every subsystem should be reusable.

Avoid hardcoding

* textures
* shaders
* meshes
* element colors
* timing

Prefer parameter-driven behavior.

A reusable system enables many different skills with minimal code changes.

# 27. Common Architecture Mistakes

Avoid

❌ Writing gameplay inside rendering code.

❌ Hardcoding every effect.

❌ Coupling Trail to a specific skill.

❌ Coupling Shader to gameplay logic.

❌ Duplicating systems instead of parameterizing them.

❌ Triggering every visual effect from one function.

❌ Updating unrelated systems together.

A clean architecture is easier to extend and produces more consistent skills.

Before considering a skill complete, verify:

□ Clear lifecycle

□ Well-defined states

□ Modular systems

□ Independent layers

□ Strong Hero Moment

□ Proper scaling by level

□ Meaningful milestone upgrades

□ Intelligent quantity scaling

A premium skill is not defined by visual complexity alone.
